# AGENTS.md

面向 OpenAI Codex 的通用编码规则。可以作为全局规则，也可以复制到具体项目根目录后与项目规则合并。

这些规则偏向谨慎而不是速度。简单拼写修复、明显的一行改动可以简化流程；非平凡任务必须先明确目标、再小步验证。

## 工作原则

- 先理解任务，再修改代码。
- 不要替用户偷偷做假设；不确定就说明不确定。
- 如果需求有多种解释，先列出可能解释和取舍，再选择最安全的最小方案；风险较高时先问。
- 优先使用项目已有框架、工具、风格和约定。
- 每次改动都应能追溯到用户请求。
- 不做无关优化、无关格式化、无关重构。
- 修改后说明改了什么、为什么这样改、如何验证。

## 修改代码前的思考流程

在实现前先完成以下检查：

1. 明确目标：这次任务要解决什么具体问题？
2. 明确边界：哪些文件、模块、行为需要改？哪些不应该碰？
3. 明确假设：是否存在输入、平台、硬件、接口、兼容性等不确定点？
4. 明确验证：改完后用什么命令、测试、构建、手工步骤证明成功？
5. 明确风险：是否可能影响启动流程、数据格式、公开 API、硬件外设、实时任务或中断？

如果无法定义成功标准，不要贸然大改。先询问，或用只读分析给出选项。

## 简单优先

- 用最少代码解决当前问题。
- 不为一次性逻辑创建抽象层。
- 不添加用户没有要求的配置项、扩展机制、策略模式或通用框架。
- 不提前实现“以后可能需要”的功能。
- 不为理论上不可能出现的场景堆叠复杂错误处理。
- 如果 50 行可以清楚解决，不要写成 200 行。

判断标准：资深工程师看到 diff 时，应能快速确认这是解决当前问题的直接改动。

## 小步修改原则

- 先做最小可验证改动，再根据结果继续。
- 优先修复根因，不用大范围绕开问题。
- 保持 diff 小而清晰。
- 修改现有代码时匹配原有命名、格式、错误处理和测试风格。
- 只清理自己改动造成的未使用导入、变量、函数或文件。
- 发现无关死代码、坏味道或潜在 bug，可以在回复中指出，但不要顺手删除。

## 不要过度设计

- 不要把简单函数改成类体系，除非项目已有模式或需求明确需要。
- 不要引入新依赖，除非现有工具无法合理完成任务。
- 不要为了“更灵活”改变调用方契约。
- 不要把局部修复扩展成架构重写。
- 不要把异常处理、日志、配置、缓存、异步化一起塞进一个小需求里。

## 不要随意重构

- 用户没有要求重构时，不重排文件结构、不批量改名、不格式化全文件。
- 修 bug 时只改与 bug 直接相关的路径。
- 加功能时只接入必要入口，不顺手改相邻功能。
- 不删除自己没充分理解的注释、兼容逻辑、延时、宏定义、硬件初始化顺序或构建脚本。
- 如果重构是完成任务的必要条件，先说明原因、范围和验证方式。

## 不要猜测需求

以下情况需要先澄清或明确说明假设：

- “优化”“更快”“更稳定”等目标没有可测指标。
- “导出”“同步”“集成”等词可能对应多种产品形态。
- 涉及数据删除、迁移、权限、隐私、收费、外部服务。
- 涉及硬件引脚、时钟、DMA、中断优先级、屏幕分辨率、触摸方向等板级信息。
- 涉及无法从仓库中确认的协议、设备型号、芯片型号或 SDK 版本。

如果风险低且可以从代码中推断，允许采用最小假设继续，但必须在回复中说明。

## 目标驱动执行

把任务转换成可验证目标：

- 修 bug：先尝试复现，再修复，再证明不再复现。
- 加验证：覆盖有效输入和无效输入。
- 改接口：确认调用方、测试和文档是否需要同步。
- 改构建：运行对应构建命令，检查产物或错误信息。
- 改嵌入式逻辑：尽量说明编译、烧录、串口、屏幕、外设现象的验证步骤。

多步骤任务使用简短计划：

```text
1. 检查现状 -> 验证: 找到相关代码和调用路径
2. 最小修改 -> 验证: 单元测试/构建通过
3. 回归检查 -> 验证: 相关功能仍正常
```

## 修改后的说明

完成后回复应包含：

- 改了哪些文件和主要行为。
- 为什么这个方案是最小可行改动。
- 已经运行了哪些验证命令，以及结果。
- 没有运行的验证要明确说明原因。
- 如有残余风险，说明风险和建议的下一步。

## 代码审查要求

当用户要求检查代码、审查代码、找 bug、看风险时：

- 先列问题，按严重程度排序。
- 每个问题给出文件和行号。
- 重点关注真实 bug、回归风险、缺失测试、边界条件和安全问题。
- 不把个人风格偏好当成缺陷。
- 如果没有发现问题，明确说没有发现，并说明仍未覆盖的验证范围。

## 调试要求

遇到编译错误、运行错误、崩溃、烧录失败、串口无输出、屏幕不显示时：

- 先收集完整错误、复现步骤、最近改动和环境信息。
- 从最小复现和最可能根因开始。
- 一次只改一个变量，避免同时改多个可能原因。
- 保留原始错误信息，不要只凭猜测修。
- 修复后给出复现前后对比和验证步骤。

## 嵌入式 / STM32 / LVGL / FreeRTOS / C 项目特别规则

如果项目涉及 STM32、HAL、LL、CMSIS、FreeRTOS、LVGL、BSP、启动文件、链接脚本、LCD、触摸屏、串口、SPI、I2C、DMA、中断或裸机 C：

- 更加保守，优先读懂初始化顺序、时钟树、引脚复用、外设句柄和中断关系。
- 不随意改启动文件、链接脚本、时钟配置、中断向量表、DMA 配置、FreeRTOS 栈大小和任务优先级。
- 不随意删除延时、缓存清理、屏障指令、`volatile`、弱符号、对齐属性和段属性。
- 修改 HAL/LVGL/FreeRTOS 相关代码时，检查线程上下文、中断上下文和锁/队列/信号量使用是否正确。
- C 代码优先保持清晰直接，谨慎使用宏魔法、全局状态和复杂回调链。
- UI 显示问题要同时考虑背光、复位脚、SPI/RGB/DSI 时序、显存地址、颜色格式、flush 回调、tick 和任务调度。
- 串口无输出要同时检查波特率、引脚复用、时钟、重定向、DMA/中断模式和是否进入 HardFault。
- 烧录失败要区分连接、供电、BOOT 引脚、读保护、下载算法、链接地址和调试器配置。
- 无法在本机验证硬件现象时，给出明确的板上验证步骤。

## 成功标志

这些规则生效时，应该看到：

- diff 更小，只包含请求相关改动。
- 代码第一次实现就足够简单。
- 不确定点在实现前被暴露，而不是出错后才发现。
- 每次修改都有清楚的验证路径。
